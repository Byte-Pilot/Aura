# Документация Backend API Aura

Базовый URL: `/api` (например, `http://localhost:<PORT>/api`)

Формат общения: JSON (`Content-Type: application/json`).
Все ответы в случае успеха возвращают соответствующий JSON, в случае ошибки — JSON формата `{"error": "текст ошибки"}` (если возвращаются через `middleware.ErrorResponse`).

---

## 🌍 Публичные эндпоинты

Эти эндпоинты не требуют авторизации и используются клиентской частью сайта.

### 1. Проверка доступности дат (Check Availability)
Проверяет, свободен ли апартамент на выбранные даты.

*   **URL:** `/availability`
*   **Метод:** `GET`
*   **Query-параметры:**
    *   `from` (строка, `YYYY-MM-DD`) — дата заезда.
    *   `to` (строка, `YYYY-MM-DD`) — дата выезда.
    *   `apartment_number` (число) — номер апартамента.
*   **Пример успешного ответа (200 OK):**
    ```json
    {
      "available": true
    }
    ```
*   **Ошибки:** `400 Bad Request` при неверных параметрах/датах в прошлом.

### 2. Создание бронирования (Create Booking)
Создает новую заявку на бронирование. Возвращает конфликт, если даты заняты.

*   **URL:** `/bookings`
*   **Метод:** `POST`
*   **Тело запроса (Body):**
    ```json
    {
      "name": "Иван Иванов",
      "phone": "+79991234567",
      "telegram": "@ivanov", 
      "apartment_number": 1,
      "check_in": "2026-05-01",
      "check_out": "2026-05-10"
    }
    ```
    *Примечание: `telegram` необязателен.*
*   **Пример успешного ответа (201 Created):**
    ```json
    {
      "message": "booking created successfully",
      "id": 123
    }
    ```
*   **Ошибки:** `400 Bad Request` при неверных данных, `409 Conflict` если даты заняты.

### 3. Получение календаря занятости (Get Calendar)
Возвращает список всех интервалов дат, когда апартамент занят (забронирован или заблокирован). Этот список нужен для блокировки дат в календаре на фронтенде пользователя.

*   **URL:** `/calendar`
*   **Метод:** `GET`
*   **Query-параметры:**
    *   `apartment_number` (число) — номер апартамента.
*   **Пример успешного ответа (200 OK):**
    ```json
    [
      {
        "check_in": "2026-03-20",
        "check_out": "2026-03-25"
      },
      {
        "check_in": "2026-04-01",
        "check_out": "2026-04-05"
      }
    ]
    ```

## 🔒 Админские эндпоинты 

Путь: `/admin/...` (полный путь с базовым: `/api/admin/...`).
Для всех методов, кроме `login` и `refresh`, требуется наличие валидного `access_token` в куках. Куки устанавливаются сервером (`httpOnly`), фронтенд должен делать запросы с `credentials: 'include'`.

### 1. Авторизация (Login)
*   **URL:** `/admin/login`
*   **Метод:** `POST`
*   **Тело запроса:**
    ```json
    {
      "username": "admin",
      "password": "yourpassword"
    }
    ```
*   **Успешный ответ (200 OK):** Сервер устанавливает `access_token` (на 15 мин) и `refresh_token` (на 7 дней) в куки.
    ```json
    {
      "message": "login successful"
    }
    ```

### 2. Обновление токена (Refresh)
*   **URL:** `/admin/refresh`
*   **Метод:** `POST`
*   **Успешный ответ (200 OK):** Сервер читает `refresh_token` из куков и устанавливает новые куки.
    ```json
    {
      "message": "tokens refreshed"
    }
    ```

### 3. Выход (Logout)
*   **URL:** `/admin/logout`
*   **Метод:** `POST`
*   *(Требует авторизации)*
*   **Успешный ответ (200 OK):** Сервер очищает куки токенов.
    ```json
    {
      "message": "logged out"
    }
    ```

### 4. Получение всех бронирований (Get All Bookings)
Позволяет админке получить весь список броней и заблокированных интервалов.

*   **URL:** `/admin/bookings`
*   **Метод:** `GET`
*   *(Требует авторизации)*
*   **Пример успешного ответа (200 OK):**
    ```json
    [
      {
        "id": 1,
        "name": "Иван Иванов",
        "phone": "+7...",
        "telegram": "@...",
        "apartment_number": 1,
        "check_in": "2026-05-01T00:00:00Z",
        "check_out": "2026-05-10T00:00:00Z",
        "status": "pending", 
        "created_at": "...",
        "updated_at": "..."
      }
    ]
    ```
    *Статусы могут быть:* `pending`, `confirmed`, `cancelled`, `blocked`.

### 5. Обновление статуса бронирования (Update Status)
Используется админом, чтобы подтвердить (`confirmed`) или отменить (`cancelled`) бронь.

*   **URL:** `/admin/bookings/{id}`
*   **Метод:** `PATCH`
*   *(Требует авторизации)*
*   **Тело запроса:**
    ```json
    {
      "status": "confirmed"
    }
    ```
*   **Успешный ответ (200 OK):**
    ```json
    {
      "message": "status updated"
    }
    ```

### 6. Ручная блокировка дат (Block Dates)
Инкапсулирует ручное закрытие продаж / ремонты админом. Создает запись со статусом `blocked`.

*   **URL:** `/admin/block`
*   **Метод:** `POST`
*   *(Требует авторизации)*
*   **Тело запроса:**
    ```json
    {
      "apartment_number": 1,
      "check_in": "2026-06-01",
      "check_out": "2026-06-15"
    }
    ```
*   **Успешный ответ (201 Created):**
    ```json
    {
      "message": "dates blocked successfully"
    }
    ```

### 7. Разблокировка дат (Unblock Dates)
Снимает ранее установленную ручную блокировку (удаляет статус `blocked` для совпавших дат).

*   **URL:** `/admin/blocked-dates`
*   **Метод:** `PATCH`
*   *(Требует авторизации)*
*   **Тело запроса:**
    ```json
    {
      "apartment_number": 1,
      "check_in": "2026-06-01",
      "check_out": "2026-06-15"
    }
    ```
*   **Успешный ответ (200 OK):**
    ```json
    {
      "message": "dates unblocked successfully"
    }
    ```
